# CDK Construct → Terraform Mapping

Use this as a cheat sheet during Phase 3 (writing Terraform). Organised by AWS service family. Each section lists:

- The CDK construct (with its C# namespace)
- The Terraform resource (AWS provider)
- Notes on property translation, caveats, and when to prefer a community/repo module instead

**Always check `references/module-inventory.md` first.** If the host repo has a module that already wraps the pattern below, use it. This reference is for when you have to go to raw resources, or when authoring a new module.

## How CDK tokens become Terraform references

CDK uses "tokens" (opaque string placeholders resolved at synth time) for cross-resource references. In the synthesized CloudFormation they appear as `{ "Ref": "..." }` or `{ "Fn::GetAtt": ["X", "Attr"] }`. In Terraform, these become direct attribute references:

| CDK / CFN | Terraform equivalent |
|---|---|
| `bucket.BucketArn` in C# | `aws_s3_bucket.this.arn` |
| `{ "Ref": "MyBucketF68F3FF0" }` in synth | `aws_s3_bucket.this.id` (usually — CFN `Ref` depends on resource type) |
| `{ "Fn::GetAtt": ["MyBucket", "Arn"] }` | `aws_s3_bucket.this.arn` |
| `{ "Fn::Sub": "arn:aws:s3:::${Bucket}" }` | `"arn:aws:s3:::${aws_s3_bucket.this.id}"` (string interpolation) |
| `{ "Fn::Join": ["/", ["arn:aws:iam::", {"Ref": "AWS::AccountId"}, ":role/X"]] }` | `"arn:aws:iam::${data.aws_caller_identity.current.account_id}:role/X"` |
| `Stack.Of(this).Account` in C# | `data.aws_caller_identity.current.account_id` |
| `Stack.Of(this).Region` in C# | `data.aws_region.current.name` |
| `props.Env.Account` | same as above, or a Terraform variable if parameterised |

CDK also generates logical IDs by concatenating construct paths (`MyStack/MyBucket/Resource` → `MyBucketF68F3FF0` after hashing). These logical IDs are *not* physical names — physical names come from the `BucketName` property (or are auto-generated by AWS if not set). When writing Terraform, use the physical name if CDK set one explicitly; otherwise let Terraform auto-name (or name it after the module/resource).

## Networking

### `Amazon.CDK.AWS.EC2.Vpc`

**CDK:**
```csharp
var vpc = new Vpc(this, "Vpc", new VpcProps {
    MaxAzs = 3,
    NatGateways = 3,
    SubnetConfiguration = new[] {
        new SubnetConfiguration { Name = "public", SubnetType = SubnetType.PUBLIC, CidrMask = 24 },
        new SubnetConfiguration { Name = "private", SubnetType = SubnetType.PRIVATE_WITH_EGRESS, CidrMask = 24 }
    }
});
```

**Prefer:** the host repo's networking module (almost every Terraform repo has one). If one doesn't exist and you need to author: use `terraform-aws-modules/vpc/aws` if the repo already uses public modules, otherwise write a new module following the repo style.

**Raw resources (if strictly needed):** `aws_vpc`, `aws_subnet` × N, `aws_internet_gateway`, `aws_nat_gateway`, `aws_eip`, `aws_route_table`, `aws_route_table_association`, `aws_route`.

**Gotchas:**
- `MaxAzs` + `NatGateways = MaxAzs` means one NAT per AZ (prod pattern). `NatGateways = 1` means one NAT shared (dev pattern). Preserve the distinction — it's a cost/HA trade-off the team made.
- `SubnetType.PRIVATE_WITH_EGRESS` needs a NAT; `SubnetType.PRIVATE_ISOLATED` doesn't. Different Terraform wiring.
- CDK auto-tags subnets with `aws-cdk:subnet-name` and `aws-cdk:subnet-type`. Terraform won't — some teams care (for aws-load-balancer-controller auto-discovery), some don't. Ask or check other modules.

### `Amazon.CDK.AWS.EC2.SecurityGroup`

**CDK:**
```csharp
var sg = new SecurityGroup(this, "Sg", new SecurityGroupProps {
    Vpc = vpc,
    Description = "...",
    AllowAllOutbound = true
});
sg.AddIngressRule(Peer.Ipv4("10.0.0.0/8"), Port.Tcp(443));
```

**Terraform:** `aws_security_group` + `aws_vpc_security_group_ingress_rule` / `aws_vpc_security_group_egress_rule` (newer rule resources; one rule per resource). Avoid inline `ingress` / `egress` blocks inside `aws_security_group` unless the repo already uses that style — the split-resource pattern is strongly preferred in 2024+.

**Gotchas:**
- `AllowAllOutbound = true` in CDK means an implicit egress rule 0.0.0.0/0 ALL. Terraform's `aws_security_group` does NOT create an egress rule by default — you must add one explicitly with `aws_vpc_security_group_egress_rule`.
- CDK's `Connections.AllowFrom` creates both sides of the rule (ingress on target, egress on source if needed). In Terraform, you write both explicitly.

### `Amazon.CDK.AWS.EC2.InterfaceVpcEndpoint` / `GatewayVpcEndpoint`

**Terraform:** `aws_vpc_endpoint`. Interface endpoints need `subnet_ids`, `security_group_ids`, `private_dns_enabled`. Gateway endpoints (S3, DynamoDB) need `route_table_ids`.

## Compute

### `Amazon.CDK.AWS.Lambda.Function`

**CDK:**
```csharp
var fn = new Function(this, "Fn", new FunctionProps {
    Runtime = Runtime.DOTNET_8,
    Handler = "MyApp::MyApp.Function::FunctionHandler",
    Code = Code.FromAsset("src/MyApp/bin/Release/net8.0/publish"),
    MemorySize = 512,
    Timeout = Duration.Seconds(30)
});
```

**Prefer:** host repo's Lambda module.

**Raw:** `aws_lambda_function`, plus `aws_iam_role` + `aws_iam_role_policy` for the execution role, plus `aws_cloudwatch_log_group` for `/aws/lambda/<name>` (CDK creates this implicitly; Terraform doesn't unless you do).

**Gotchas:**
- `Code.FromAsset("<path>")` implies CDK builds and uploads the zip at `cdk deploy` time. In Terraform you either (a) use `archive_file` data source to zip and `aws_s3_object` to upload, or (b) build the zip externally in CI and pass its S3 location as a variable. The team's preference shows up in existing Lambda modules — match it.
- `Runtime.DOTNET_8` → `"dotnet8"` in Terraform.
- CDK auto-creates the log group with retention defaults from context; Terraform needs an explicit `aws_cloudwatch_log_group` with `retention_in_days`.

### `Amazon.CDK.AWS.ECS.FargateService` / `FargateTaskDefinition`

**CDK:**
```csharp
var task = new FargateTaskDefinition(this, "Task", new FargateTaskDefinitionProps {
    Cpu = 512,
    MemoryLimitMiB = 1024
});
task.AddContainer("App", new ContainerDefinitionOptions {
    Image = ContainerImage.FromRegistry("nginx:latest"),
    PortMappings = new[] { new PortMapping { ContainerPort = 80 } }
});
var svc = new FargateService(this, "Svc", new FargateServiceProps {
    Cluster = cluster,
    TaskDefinition = task,
    DesiredCount = 2
});
```

**Terraform:** `aws_ecs_task_definition`, `aws_ecs_service`, with `aws_iam_role` for execution and task roles. Container definitions go into the `container_definitions` JSON string — produce it via `jsonencode({ ... })` in a `locals` block or a separate file.

**Gotchas:**
- CDK sets sensible defaults (log driver `awslogs` to `/ecs/<service>`); Terraform needs these spelled out. See the `logConfiguration` inside container_definitions.
- `ContainerImage.FromEcrRepository` in CDK implies the image lives in a CDK-managed ECR repo. In Terraform, pass the image URI via variable.
- `FargateService.AttachToApplicationTargetGroup(...)` in CDK creates the load_balancer wiring. In Terraform, use the `load_balancer` block inside `aws_ecs_service`.

### `Amazon.CDK.AWS.ElasticLoadBalancingV2.ApplicationLoadBalancer`

**CDK:**
```csharp
var alb = new ApplicationLoadBalancer(this, "Alb", new ApplicationLoadBalancerProps {
    Vpc = vpc,
    InternetFacing = true
});
var listener = alb.AddListener("Listener", new BaseApplicationListenerProps {
    Port = 443,
    Certificates = new[] { ListenerCertificate.FromArn(certArn) }
});
listener.AddTargets("Targets", new AddApplicationTargetsProps {
    Port = 80,
    Targets = new[] { service }
});
```

**Terraform:** `aws_lb` (type `"application"`), `aws_lb_listener`, `aws_lb_target_group`, `aws_lb_listener_rule` (for more complex routing). Wire the service to the target group via the `load_balancer` block in `aws_ecs_service`.

**Gotchas:**
- CDK's `listener.AddTargets(...)` auto-creates the target group and attaches targets. In Terraform you do each explicitly.
- `InternetFacing = true` → `internal = false` (note the inversion).
- Certificates: CDK `ListenerCertificate.FromArn` → Terraform `certificate_arn` on the listener, plus optional `aws_lb_listener_certificate` resources for SNI extras.

## Storage

### `Amazon.CDK.AWS.S3.Bucket`

**Terraform:** `aws_s3_bucket` + several satellite resources. AWS moved S3 config *out* of the main bucket resource in 2022 — you now need separate resources for:

| CDK property | Terraform resource |
|---|---|
| `Encryption = BucketEncryption.S3_MANAGED` | `aws_s3_bucket_server_side_encryption_configuration` |
| `Versioned = true` | `aws_s3_bucket_versioning` |
| `BlockPublicAccess = BlockPublicAccess.BLOCK_ALL` | `aws_s3_bucket_public_access_block` |
| `LifecycleRules = [...]` | `aws_s3_bucket_lifecycle_configuration` |
| `Cors = [...]` | `aws_s3_bucket_cors_configuration` |
| `Website = ...` | `aws_s3_bucket_website_configuration` |

If the host repo has an `s3-bucket` module, use it — this is exactly the case where a module saves a lot of boilerplate.

**Gotchas:**
- `BucketName` is optional in CDK (AWS generates one). If not set, do **not** set `bucket` in Terraform either (or use `bucket_prefix`) — otherwise you'll get name collisions across environments.
- `AutoDeleteObjects = true` in CDK does something magical via a custom resource. Terraform has no equivalent — use `force_destroy = true` on the bucket, which enables deletion of non-empty buckets but only at destroy time.

### `Amazon.CDK.AWS.DynamoDB.Table`

**Terraform:** `aws_dynamodb_table`. Straightforward mapping. `BillingMode.PAY_PER_REQUEST` → `billing_mode = "PAY_PER_REQUEST"`. `BillingMode.PROVISIONED` requires `read_capacity` + `write_capacity`.

**Gotchas:**
- `PointInTimeRecovery = true` in CDK → `point_in_time_recovery { enabled = true }` block in Terraform.
- CDK's stream specification (`StreamSpecification = StreamViewType.NEW_AND_OLD_IMAGES`) → `stream_enabled = true` + `stream_view_type = "NEW_AND_OLD_IMAGES"`.

### `Amazon.CDK.AWS.RDS.DatabaseInstance` / `DatabaseCluster`

**Terraform:** `aws_db_instance` / `aws_rds_cluster` + `aws_rds_cluster_instance`. Plus `aws_db_subnet_group`, `aws_security_group`, `aws_secretsmanager_secret` (if using managed master password).

**Gotchas:**
- **RDS is stateful** — default to `import` (not greenfield) when converting an existing deployment. Data loss otherwise.
- CDK's `CredentialsFromGeneratedSecret` manages a Secrets Manager secret with rotation. Terraform doesn't do this in one step — either use `aws_rds_cluster.manage_master_user_password = true` (AWS-managed) or wire `aws_secretsmanager_secret` + `aws_secretsmanager_secret_rotation` explicitly.

## IAM

### `Amazon.CDK.AWS.IAM.Role`

**CDK:**
```csharp
var role = new Role(this, "Role", new RoleProps {
    AssumedBy = new ServicePrincipal("lambda.amazonaws.com"),
    ManagedPolicies = new[] {
        ManagedPolicy.FromAwsManagedPolicyName("service-role/AWSLambdaBasicExecutionRole")
    }
});
role.AddToPolicy(new PolicyStatement(new PolicyStatementProps {
    Actions = new[] { "s3:GetObject" },
    Resources = new[] { bucket.ArnForObjects("*") }
}));
```

**Terraform:** `aws_iam_role` + `aws_iam_role_policy_attachment` (for managed policies) + `aws_iam_role_policy` (for inline policies, via `jsonencode(...)` or `aws_iam_policy_document` data source).

**Gotchas:**
- `ServicePrincipal("lambda.amazonaws.com")` → assume_role_policy with `Service = "lambda.amazonaws.com"`.
- `ManagedPolicy.FromAwsManagedPolicyName("X")` → `policy_arn = "arn:aws:iam::aws:policy/X"`.
- Prefer `data.aws_iam_policy_document` over raw `jsonencode` for policy documents — matches what most Terraform repos use, and handles the formatting quirks AWS is picky about.
- IAM policies are the #1 source of `terraform plan` drift on import. The JSON AWS stores back is reformatted; expect to tweak statement ordering and whitespace until plan is clean.

## Observability

### `Amazon.CDK.AWS.Logs.LogGroup`

**Terraform:** `aws_cloudwatch_log_group`. `RetentionDays = RetentionDays.ONE_MONTH` → `retention_in_days = 30`.

### `Amazon.CDK.AWS.CloudWatch.Alarm`

**Terraform:** `aws_cloudwatch_metric_alarm`. Property-for-property mapping; watch for `Metric.With(...)` chains in CDK — they all collapse to properties on the one Terraform resource.

## Messaging

### `Amazon.CDK.AWS.SQS.Queue`

**Terraform:** `aws_sqs_queue`. DLQ wiring: CDK's `DeadLetterQueue = { Queue = dlq, MaxReceiveCount = 3 }` → Terraform `redrive_policy = jsonencode({ deadLetterTargetArn = aws_sqs_queue.dlq.arn, maxReceiveCount = 3 })`.

### `Amazon.CDK.AWS.SNS.Topic`

**Terraform:** `aws_sns_topic`. Subscriptions are separate: `aws_sns_topic_subscription`.

### `Amazon.CDK.AWS.Events.Rule`

**Terraform:** `aws_cloudwatch_event_rule` + `aws_cloudwatch_event_target`. Note CDK's `RuleTargetInput.FromObject(...)` → Terraform `input = jsonencode({...})`.

## Secrets & config

### `Amazon.CDK.AWS.SecretsManager.Secret`

**Terraform:** `aws_secretsmanager_secret` + `aws_secretsmanager_secret_version`. Note: CDK's `SecretStringGenerator` generates a random value at deploy time; Terraform has `random_password` + the secret resource. **Never commit the generated value** — store it in the secret, not in state outputs.

### `Amazon.CDK.AWS.SSM.StringParameter`

**Terraform:** `aws_ssm_parameter`. Watch for `Type = ParameterType.SECURE_STRING` → `type = "SecureString"` + `key_id` for the KMS key.

## KMS

### `Amazon.CDK.AWS.KMS.Key`

**Terraform:** `aws_kms_key` + optionally `aws_kms_alias`. **KMS keys are stateful** — default to import, never greenfield-recreate a key that's used to encrypt existing data.

## Putting it together

When you hit a CDK construct not listed here, the pattern is always:

1. Look at what the construct *does* in CloudFormation (check `cdk synth` output — `Type` field on the resource).
2. Find the `aws_<service>` resource in the Terraform AWS provider docs with the same CFN type (usually the name is similar).
3. Map properties. CFN uses PascalCase; Terraform uses snake_case.
4. Check the repo's module inventory to see if the pattern is already wrapped.
5. If not, decide module-vs-raw based on reuse potential (Phase 2 of the convert workflow).

When in doubt, write less abstraction. A well-commented raw resource is easier to review than a clever module with the wrong boundaries.
