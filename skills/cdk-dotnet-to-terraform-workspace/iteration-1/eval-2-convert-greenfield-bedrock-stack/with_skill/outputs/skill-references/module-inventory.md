### `modules/network`

- **Purpose:** VPC with public + private subnets spread across configurable AZs, Internet Gateway, and one NAT Gateway per AZ (HA).
- **Key inputs:** `name` (string, required), `cidr_block` (string, required), `availability_zones` (list(string), required), `tags` (map(string), default `{}`).
- **Key outputs:** `vpc_id`, `vpc_cidr_block`, `public_subnet_ids` (list), `private_subnet_ids` (list), `default_security_group_id`.
- **Matches CDK constructs:** `Amazon.CDK.AWS.EC2.Vpc` with `SubnetConfiguration` for `PUBLIC` + `PRIVATE_WITH_EGRESS` and `NatGateways = <n>`.
- **Notes:** Module does **not** create VPC endpoints (interface or gateway). Callers add `aws_vpc_endpoint` resources themselves — e.g. a `bedrock-runtime` endpoint goes in the root config, not inside this module. Subnet CIDRs are computed via `cidrsubnet(cidr, 8, idx)` for public and `cidrsubnet(cidr, 8, idx + len(azs))` for private; the module doesn't let the caller override them.

### `modules/alb`

- **Purpose:** Application load balancer with HTTPS listener (default 404), HTTP→HTTPS redirect listener, and a dedicated front security group. Target groups + listener rules are intentionally outside the module — callers own those.
- **Key inputs:** `name` (string, required), `vpc_id` (string, required), `public_subnet_ids` (list(string), required), `certificate_arn` (string, required), `allowed_cidr_blocks` (list(string), default `["0.0.0.0/0"]`), `internal` (bool, default `false`), `idle_timeout` (number, default `60`), `enable_deletion_protection` (bool, default `false`), `tags` (map(string), default `{}`).
- **Key outputs:** `alb_arn`, `alb_dns_name`, `alb_zone_id`, `https_listener_arn`, `security_group_id`.
- **Matches CDK constructs:** `Amazon.CDK.AWS.ElasticLoadBalancingV2.ApplicationLoadBalancer` + HTTPS listener + HTTP→HTTPS redirect listener.
- **Notes:** SSL policy is hardcoded to `ELBSecurityPolicy-TLS13-1-2-2021-06` (matches CDK's `SslPolicy.TLS13_12`). If a service needs a different default action than 404 (e.g. forward-to-default-target-group), add a listener rule after the module instantiation pointing at the module's `https_listener_arn`. Service SGs should allow ingress from `security_group_id`.

### `modules/ecs-service`

- **Purpose:** Fargate service behind an existing ALB target group. Creates task definition, ECS service, execution + task IAM roles, CloudWatch log group (`/ecs/<service_name>`), and task security group with ALB ingress rule.
- **Key inputs:** `service_name` (string, required), `cluster_arn` (string, required), `image` (string, required), `vpc_id` (string, required), `subnet_ids` (list(string), required), `alb_security_group_id` (string, required), `target_group_arn` (string, required), `cpu` (number, default `512`), `memory` (number, default `1024`), `desired_count` (number, default `2`), `container_port` (number, default `8080`), `environment_variables` (map(string), default `{}`), `secret_arns` (map(string), default `{}`), `additional_task_role_policy_arns` (list(string), default `[]`), `log_retention_days` (number, default `30`), `tags` (map(string), default `{}`).
- **Key outputs:** `service_arn`, `service_name`, `task_definition_arn`, `task_role_arn`, `task_role_name`, `execution_role_arn`, `log_group_name`, `security_group_id`.
- **Matches CDK constructs:** `Amazon.CDK.AWS.ECS.FargateService` + `FargateTaskDefinition` + a single-container `AddContainer(...)` registered as `LoadBalancers[0]` target. Log group matches the CDK `LogGroup` pattern when named `/ecs/<service_name>`.
- **Notes:** Task role permissions for Bedrock (or similar) can be wired three ways per the module README: (1) `additional_task_role_policy_arns` for managed/customer-policy ARNs, (2) `aws_iam_role_policy` at the caller using `task_role_name`, (3) author the task role separately via `iam-role` module — the README flags this as "future" but option 2 is the path for inline-JSON per-service policies today. Lifecycle ignores `desired_count` to let autoscalers manage it. Retention defaults to 30 days, matching the CDK `RetentionDays.ONE_MONTH`. Container insights is a cluster-level setting and is not controlled by this module.

### `modules/iam-role`

- **Purpose:** Stand-alone IAM role with trust policy JSON, optional managed-policy attachments, optional keyed inline policies.
- **Key inputs:** `name` (string, required), `trust_policy_json` (string, required), `managed_policy_arns` (list(string), default `[]`), `inline_policies` (map(string), default `{}` — keyed by policy name, value is JSON), `max_session_duration` (number, default `3600`), `tags` (map(string), default `{}`).
- **Key outputs:** `role_arn`, `role_name`, `role_id`.
- **Matches CDK constructs:** `Amazon.CDK.AWS.IAM.Role` — managed policies map to `managed_policy_arns`; `AddToPolicy(...)` calls map to `inline_policies` entries (one statement-set per entry).
- **Notes:** The README explicitly documents a Bedrock task-role pattern using `data.aws_iam_policy_document.bedrock_invoke`. This is the "correct" pattern for a role that needs inline Bedrock permissions; however it replaces the task role built in to the `ecs-service` module, so using it means either not passing the role in (future feature) or using `aws_iam_role_policy` attached at the caller against `ecs-service`'s built-in `task_role_name` output. For the current conversion we'll use the latter (caller-attached `aws_iam_role_policy`) since the `ecs-service` module already builds the task role and the Bedrock policy is a single inline policy.

## How convert mode uses this file

For each resource or construct in the CDK project:

1. Read the inventory.
2. Find a module whose **key inputs** can be supplied from what the CDK construct has, and whose **key outputs** expose what downstream constructs need.
3. If a match exists → use that module.
4. If no match exists → author a new module in the same style. Add it to this inventory.
5. If the pattern is one-off / trivially small → raw resources in the root config are fine.
