## 1. Repo layout

- **Terraform root(s):** `terraform-host/` (single tree)
- **Multiple trees?** No.
- **Which tree does this skill target?** The single `terraform-host/` tree.

## 2. Module layout

- **Module directory:** `terraform-host/modules/`
- **Module-naming convention:** kebab-case, no prefix. Names reflect purpose (`network`, `alb`, `ecs-service`, `iam-role`).
- **How modules are consumed:** relative `source = "../../../modules/<name>"` path from `live/<env>/<app>/` root configs.
- **Module versioning:** none — monorepo, always latest.

## 3. File layout inside a module

- **Standard files:** `main.tf`, `variables.tf`, `outputs.tf`, `versions.tf`, `README.md`. No separate `locals.tf`.
- **What goes in `main.tf`:** `locals { common_tags = merge(var.tags, { Module = "<name>", Name = var.name }) }` at the top, followed by resources. Data sources live at their point of use in `main.tf`.
- **Use of `locals.tf`:** N/A — all locals are inline in `main.tf` at the top of the file.
- **`versions.tf` contents:** `terraform { required_version = ">= 1.5, < 2.0"; required_providers { aws = { source = "hashicorp/aws", version = "~> 5.70" } } }`. No `provider` block (modules don't declare providers).
- **README per module?** Yes. Hand-written Markdown with sections `# <name>`, short description, `## Inputs`, `## Outputs`, and optional usage-pattern blocks.

## 4. Root / live config layout

- **Environments directory:** `live/<env>/<app>/` (e.g. `live/dev/reference-service/`).
- **Environment split pattern:** folder-per-env with per-app subfolders. No Terraform workspaces.
- **File organization in root:** single `main.tf` calling modules and declaring any raw resources (target groups, listener rules, ECS cluster, additional IAM policies). Variables in `variables.tf`, outputs in `outputs.tf`, backend + provider + version pins in `versions.tf`.

## 5. Naming conventions

- **Variables:** `snake_case`. Always has `description` and `type`. Required vars have no default; optional vars always have a default.
- **Outputs:** `snake_case` with `_arn`, `_id`, `_name`, `_dns_name` suffixes. Collection outputs end in `_ids`.
- **Resource labels:** `resource "aws_X" "this"` — always `"this"` for the primary/singleton. Multiples use `for_each` (never `count` observed).
- **Module instance names:** short purpose-based (`module "network"`, `module "alb"`, `module "api_service"`).
- **Physical resource naming:** explicit names set where AWS supports them, with pattern `${var.application}-${var.environment}-<suffix>` in root configs, and `${var.name}-<suffix>` inside modules (e.g. `${var.name}-alb`, `${var.name}-igw`, `${var.name}-nat-${each.key}`).

## 6. Tagging

- **Source of tags:** BOTH `default_tags` on the provider AND an explicit `tags` variable passed into every module (because `default_tags` doesn't propagate into `container_definitions` JSON and similar).
- **Required keys:** `Environment`, `Application`, `ManagedBy = "terraform"`. Root configs also add `CostCenter`. Modules add `Module` (module name) and `Name` inside their `local.common_tags`.
- **How modules accept extra tags:** every module has `variable "tags" { type = map(string); default = {} }` and merges via `locals.common_tags = merge(var.tags, { Module = "<name>", Name = var.name })`.
- **Are tags set on every resource type that supports them?** Yes. Exceptions (like `aws_route_table_association`, which doesn't support tags) are simply untagged.

## 7. Provider configuration

- **Provider declaration location:** root configs only. Modules declare only `required_providers` in `versions.tf`.
- **`default_tags` used?** Yes, in root configs. Contents: `Environment`, `Application`, `ManagedBy = "terraform"`.
- **`aws` provider version floor:** `~> 5.70`.
- **Multiple provider instances / aliases?** None observed — single provider per root config.
- **Assume-role or profile config pattern:** not declared in observed files (assumed to come from env vars / CI-configured credentials).

## 8. Backend configuration

- **Backend type:** `s3` partial backend.
- **State key pattern:** `<env>/<app>/terraform.tfstate` (e.g. `dev/reference-service/terraform.tfstate`).
- **Lock mechanism:** DynamoDB table supplied at `terraform init` time via `-backend-config`.
- **Backend config source:** hardcoded `key = "..."` in the backend block; bucket, region, DynamoDB table supplied as partial backend config at init time.

## 9. Cross-module / cross-stack references

- **Pattern:** module outputs within a single root config. Cross-root-config references, where they would exist, use SSM parameters (not `data.terraform_remote_state`).
- **SSM parameter naming (if used):** not observed in samples. If introduced, suggest `/<env>/<app>/<resource-type>/<name>`.
- **Remote state key pattern (if used):** N/A — SSM is the cross-config transport.

## 10. Terraform version floor

- **`required_version`:** `>= 1.5, < 2.0`.
- **Uses `import` blocks?** Not observed in the sample; Terraform 1.5+ `import` blocks would be supported when needed.
- **Uses `for_each` on modules?** Not observed, but `for_each` on resources is common.
- **Uses `moved` blocks?** Not observed.

## 11. Secrets handling

- **Where secrets live:** `ecs-service` module accepts `secret_arns` (map of env-var name → Secrets Manager / SSM ARN). Secret values are never in state; resources only reference ARNs.
- **How Terraform reads them (if at all):** never read — task `container_definitions` has a `secrets` array that ECS resolves at task start.
- **Never-in-state policy:** yes, implicitly — Terraform only ever handles ARNs of secrets, not their values.

## 12. Documentation conventions

- **Module README format:** hand-written Markdown. Title `# <module-name>`, 1–2 sentence description, `## Inputs` (bullets with type + default), `## Outputs` (bullets with description), optional usage-pattern code block for non-obvious cases.
- **Inline comment style:** sparse — used to explain non-obvious wiring (e.g. why a lifecycle block ignores `desired_count`).
- **CHANGELOG / release notes per module?** None observed.

## 13. Inconsistencies noted during init

None observed. The four modules (`network`, `alb`, `ecs-service`, `iam-role`) and the one reference live config are internally consistent.

## 14. Open questions / deferred decisions

- **Container insights level.** The CDK stack uses `ContainerInsightsV2 = ContainerInsights.ENABLED` (enhanced container insights). The repo's `ecs-service` module does not expose this as a variable; it's a cluster-level setting and the cluster is declared inline in root configs. We'll set `setting { name = "containerInsights"; value = "enhanced" }` on the cluster directly in the bedrock-chat root config and surface this as a TODO for the team to formalise if cluster creation ever moves into a module.
- **ALB `allowed_cidr_blocks`.** The reference service relies on the module default `["0.0.0.0/0"]`. The CDK stack also opens 0.0.0.0/0, so we mirror that — but it's a candidate to tighten at the team's discretion.
