## 1. Repo layout

- **Terraform root(s):** `terraform-host/live/<env>/<app>/` (root/live configs), `terraform-host/modules/<name>/` (reusable modules).
- **Multiple trees?** No. A single Terraform tree rooted at `terraform-host/`.
- **Which tree does this skill target?** `terraform-host/` â€” CDK output lands as a new `live/<env>/<app>/` root config plus any new modules under `modules/`.

## 2. Module layout

- **Module directory:** `terraform-host/modules/`.
- **Module-naming convention:** kebab-case, no prefix (e.g. `network`, `alb`, `ecs-service`, `iam-role`). Names describe the *thing*, not the service (no `aws-` prefix).
- **How modules are consumed:** relative `source = "../../../modules/<name>"` from `live/<env>/<app>/`. No version pinning â€” monorepo, always-latest.
- **Module versioning:** None.

## 3. File layout inside a module

- **Standard files:** `main.tf`, `variables.tf`, `outputs.tf`, `versions.tf`, `README.md`. All five present in every module reviewed.
- **What goes in `main.tf`:** `locals` block at the top (module-owned `common_tags` merge), then resources. Data sources live inline in `main.tf` next to the resources that use them (e.g. `data "aws_region" "current"` in `ecs-service`).
- **Use of `locals.tf`:** N/A â€” locals are inlined at the top of `main.tf`.
- **`versions.tf` contents:** `terraform { required_version = ">= 1.5, < 2.0"; required_providers { aws = { source = "hashicorp/aws"; version = "~> 5.70" } } }`. No provider block.
- **README per module?** Yes. Hand-written Markdown. Sections: one-line intro, `## Inputs`, `## Outputs`, and sometimes `## Pattern: <name>` or `## Extending ...` for usage recipes.

## 4. Root / live config layout

- **Environments directory:** `terraform-host/live/<env>/<app>/`. Example: `live/dev/reference-service/`.
- **Environment split pattern:** Folder per env (`live/dev/`, implicitly `live/stg/`, `live/prd/`).
- **File organization in root:** `main.tf`, `variables.tf`, `outputs.tf`, `versions.tf`. `main.tf` holds a top-level `locals { common_tags = ... }` block plus module calls and any raw resources (e.g. `aws_ecs_cluster`, `aws_lb_target_group`, `aws_lb_listener_rule`). No subsystem-split files.

## 5. Naming conventions

- **Variables:** `snake_case`. `description` and `type` always set. Required vars omit `default`; optional vars provide one. `environment` and `application` have `default` values in the example â€” confirmed as a per-env tfvars override pattern.
- **Outputs:** `snake_case` with `_arn`, `_id`, `_name`, `_dns_name` suffixes. Collection outputs use `_ids` (e.g. `public_subnet_ids`). Every output has `description`.
- **Resource labels:** primary resource uses `"this"` (e.g. `resource "aws_vpc" "this"`). Multiples use `for_each` with a descriptive key (e.g. `for_each = { for idx, az in var.availability_zones : az => idx }`).
- **Module instance names:** match the module purpose in the caller (`module "network"`, `module "alb"`, `module "api_service"`). Service-instance modules may take an app-specific name.
- **Physical resource naming:** explicit `name` properties set with the pattern `${var.application}-${var.environment}[-<suffix>]`. No reliance on AWS auto-naming.

## 6. Tagging

- **Source of tags:** BOTH `default_tags` on the provider AND a `local.common_tags` map explicitly passed to every module and tagged on every in-root resource. The README comment is explicit: `default_tags` doesn't propagate into `container_definitions` JSON, so the explicit pass-through matters.
- **Required keys:** `Environment`, `Application`, `CostCenter`, `ManagedBy`. Modules additionally stamp `Module = "<module-name>"` and `Name = "<var.name>"` via `local.common_tags = merge(var.tags, { Module = "...", Name = var.name })`.
- **How modules accept extra tags:** `variable "tags" { type = map(string); default = {} }` merged with the module's `common_tags` locals.
- **Are tags set on every resource type that supports them?** Yes. `aws_vpc`, subnets, IGW, NAT GWs, EIPs, route tables, ALBs, listeners, SGs, log groups, ECS cluster / task / service, IAM roles â€” all tagged. Rule-level SG resources (`aws_vpc_security_group_ingress_rule`) are not tagged in the example (matches common practice â€” they don't sprout tags usefully).

## 7. Provider configuration

- **Provider declaration location:** Root configs only. Modules never declare `provider "aws"`; they only list `required_providers` in `versions.tf`.
- **`default_tags` used?** Yes â€” on the provider in the root config, with `Environment`, `Application`, `ManagedBy`.
- **`aws` provider version floor:** `~> 5.70`.
- **Multiple provider instances / aliases?** None observed in reviewed configs.
- **Assume-role or profile config pattern:** Not explicit in example; inferred to be CI-driven (credentials injected outside Terraform).

## 8. Backend configuration

- **Backend type:** `s3` (partial).
- **State key pattern:** `<env>/<app>/terraform.tfstate` (e.g. `dev/reference-service/terraform.tfstate`).
- **Lock mechanism:** DynamoDB lock table supplied at `terraform init -backend-config=...`.
- **Backend config source:** Partial backend â€” only `key` is hardcoded in `versions.tf`; `bucket`, `region`, `dynamodb_table` are supplied at init time via `-backend-config`.

## 9. Cross-module / cross-stack references

- **Pattern:** Module outputs consumed directly within a root config. Cross-root references use SSM parameters (per README), not `data.terraform_remote_state`.
- **SSM parameter naming (if used):** Not yet exercised in the reference config; no explicit naming convention documented.
- **Remote state key pattern (if used):** N/A â€” explicitly avoided in favour of SSM.

## 10. Terraform version floor

- **`required_version`:** `">= 1.5, < 2.0"` â€” same everywhere.
- **Uses `import` blocks?** Not in the reference config, but the version floor (>= 1.5) permits them. Convert mode adopts this pattern for any must-not-recreate resources.
- **Uses `for_each` on modules?** Not observed in the example. Module iteration is done at the resource level inside modules (public/private subnets via `for_each`).
- **Uses `moved` blocks?** Not observed.

## 11. Secrets handling

- **Where secrets live:** Secrets Manager / SSM SecureString (inferred from the `ecs-service` module's `secret_arns` variable, which accepts `key -> ARN` pairs feeding the task definition `secrets` array).
- **How Terraform reads them (if at all):** Terraform references ARNs only; actual secret values are resolved by ECS agents at task launch. Terraform does not read secret values.
- **Never-in-state policy:** Implicit â€” secrets are always by ARN; no resource in reviewed code reads secret text into state.

## 12. Documentation conventions

- **Module README format:** Hand-written. Intro paragraph + `## Inputs` + `## Outputs` + optional `## Pattern:` recipe. Input list uses a bulleted `- \`name\` (type, required/default value) - description` format.
- **Inline comment style:** Sparse. Used to justify non-obvious choices (e.g. the `default_tags`-doesn't-propagate note in the repo README, the `lifecycle { ignore_changes = [desired_count] }` justification in `ecs-service`).
- **CHANGELOG / release notes per module?** No.

## 13. Inconsistencies noted during init

None observed. The four modules are stylistically identical (file layout, locals, tagging, naming, README shape). The reference-service root matches the module style.

## 14. Open questions / deferred decisions

- **Region:** README declares `eu-central-1` as default. Root provider hardcodes it. Root variables do not expose a `region` override. Matches CDK's `cdk.json` context `"region": "eu-central-1"`.

- **VPC endpoints.** `network` module does **not** create VPC endpoints. Interface endpoints (e.g. `bedrock-runtime`) are expected at the call site as raw `aws_vpc_endpoint` resources â€” matches the inventory template note.
