## `modules/network`

- **Purpose:** VPC with public + private subnets across configurable AZs, plus IGW, per-AZ NAT gateway + EIP, and public/private route tables with associations.
- **Key inputs:** `name` (string, required), `cidr_block` (string, required), `availability_zones` (list(string), required), `tags` (map(string), default `{}`).
- **Key outputs:** `vpc_id`, `vpc_cidr_block`, `public_subnet_ids` (list), `private_subnet_ids` (list), `default_security_group_id`.
- **Matches CDK constructs:** `Amazon.CDK.AWS.EC2.Vpc` with `SubnetConfiguration` for `PUBLIC` + `PRIVATE_WITH_EGRESS` (or similar). 1 NAT per AZ matches `NatGateways = <len(azs)>`.
- **Notes:**
  - Does **not** create VPC endpoints. Callers add interface / gateway endpoints at the call site as raw `aws_vpc_endpoint` resources.
  - CIDR allocation is fixed: `cidrsubnet(var.cidr_block, 8, i)` for public, `cidrsubnet(var.cidr_block, 8, i + len(azs))` for private. If CDK's synthesised subnet CIDRs differ from this formula, callers may need to diverge (or import subnets to the computed addresses).
  - Uses `for_each` keyed by AZ name throughout. Imports for subnets / NAT GWs / route tables need the bracketed key syntax: e.g. `module.network.aws_subnet.public["eu-central-1a"]`.

## `modules/alb`

- **Purpose:** Application load balancer with an HTTPS listener (default 404 fixed-response), HTTP->HTTPS redirect listener, and a front security group. Target groups and listener rules are intentionally NOT inside this module.
- **Key inputs:** `name` (string, required), `vpc_id` (string, required), `public_subnet_ids` (list(string), required), `certificate_arn` (string, required), `allowed_cidr_blocks` (list(string), default `["0.0.0.0/0"]`), `internal` (bool, default `false`), `idle_timeout` (number, default `60`), `enable_deletion_protection` (bool, default `false`), `tags` (map(string), default `{}`).
- **Key outputs:** `alb_arn`, `alb_dns_name`, `alb_zone_id`, `https_listener_arn`, `security_group_id`.
- **Matches CDK constructs:** `Amazon.CDK.AWS.ElasticLoadBalancingV2.ApplicationLoadBalancer` + `AddListener(443, HTTPS)` + `AddListener(80, redirect to 443)` + an `AlbSg` security group with 443/80 ingress.
- **Notes:**
  - TLS policy is hardcoded `ELBSecurityPolicy-TLS13-1-2-2021-06`. CDK's `SslPolicy.TLS13_12` maps to this same policy.
  - Default action on HTTPS is a fixed `404`. Callers wire their target groups via separate `aws_lb_listener_rule` resources pointed at `module.alb.https_listener_arn`.

## `modules/ecs-service`

- **Purpose:** Fargate service + task definition + execution / task IAM roles + CloudWatch log group (`/ecs/<service>`) + task-level security group. Registers itself against an existing ALB target group.
- **Key inputs:** `service_name` (string, required), `cluster_arn` (string, required), `image` (string, required), `vpc_id` (string, required), `subnet_ids` (list(string), required), `alb_security_group_id` (string, required), `target_group_arn` (string, required), `cpu` (number, default `512`), `memory` (number, default `1024`), `desired_count` (number, default `2`), `container_port` (number, default `8080`), `environment_variables` (map(string), default `{}`), `secret_arns` (map(string), default `{}`), `additional_task_role_policy_arns` (list(string), default `[]`), `log_retention_days` (number, default `30`), `tags` (map(string), default `{}`).
- **Key outputs:** `service_arn`, `service_name`, `task_definition_arn`, `task_role_arn`, `task_role_name`, `execution_role_arn`, `log_group_name`, `security_group_id`.
- **Matches CDK constructs:** `Amazon.CDK.AWS.ECS.FargateTaskDefinition` + `FargateService` + `AwsLogDriver` + the `Service/SecurityGroup` CDK creates automatically. The ALB target attachment is done outside the module (caller creates `aws_lb_target_group` and passes its ARN).
- **Notes:**
  - `lifecycle { ignore_changes = [desired_count] }` on the service — autoscaling compatibility.
  - Container definition is built in a `local.container_definition` and wrapped in `jsonencode([...])`. CDK's task definition container environment list is a map(string) -> list-of-`{name, value}` transform.
  - `additional_task_role_policy_arns` is the first-choice extension point. For inline-policy needs (e.g. Bedrock invoke), attach an `aws_iam_role_policy` at the caller referencing `module.<name>.task_role_name`. The README suggests this explicitly.
  - CDK's `TaskRoleDefaultPolicy` (auto-created when you call `taskRole.AddToPolicy(...)`) maps naturally to a caller-side `aws_iam_role_policy`.

## `modules/iam-role`

- **Purpose:** Stand-alone IAM role with trust policy, managed-policy attachments, and keyed inline policies.
- **Key inputs:** `name` (string, required), `trust_policy_json` (string, required), `managed_policy_arns` (list(string), default `[]`), `inline_policies` (map(string), default `{}`), `max_session_duration` (number, default `3600`), `tags` (map(string), default `{}`).
- **Key outputs:** `role_arn`, `role_name`, `role_id`.
- **Matches CDK constructs:** `Amazon.CDK.AWS.IAM.Role` in its own right (not the implicit task/execution roles that `ecs-service` already provisions). Good fit when a role is its own thing: agent roles, Lambda execution roles outside the `lambda` module, Bedrock agent resource roles.
- **Notes:** The README demonstrates a Bedrock-access pattern for reference.

## How convert mode uses this file

For each resource or construct in the CDK project:

1. Read this inventory.
2. Find a module whose key inputs can be supplied from what the CDK construct has, and whose key outputs expose what downstream constructs need.
3. If a match exists -> use that module. If the match is partial -> consider whether extending the module makes sense, ask the user.
4. If no match exists -> author a new module in the same style. Add it to this inventory.
5. If the pattern is one-off / trivially small (a single VPC endpoint, a single inline IAM policy) -> raw resources in the root config are fine.

The guiding principle: **same inputs + outputs + purpose = same module**.
